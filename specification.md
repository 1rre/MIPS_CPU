<h1><a id="user-content-overall-goals" class="anchor" aria-hidden="true" href="#overall-goals"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Overall goals</h1>
<p>Your overall goals are to develop a working synthesisable MIPS-compatible CPU.
This CPU will interface with the world using a memory-mapped bus, which gives
it access to memory and other peripherals.</p>
<p>The goal of this coursework is not to get a single circuit working in a single
piece of hardware. Instead it is to develop a piece of IP which could be
sold and distributed to many clients, allowing them to integrate you CPU
into any number of products.  As a consequence the emphasis is on producing
a production quality CPU with a robust testing process - you should deliver
something that you expect to work on any FPGA or ASIC, rather than something
that just works on a single device.</p>
<p>The emphasis on creating a "real" CPU makes this a more complex task
than implementing a toy CPU with lots of extra debug hooks. In particular,
the emphasis on memory-based input/output is very realistic, but means
you need to be very methodical and analytical in the way you develop
both your CPU <em>and</em> your test-bench and test-cases.</p>

<p>You have a choice of two different interface styles for your CPU to support:</p>
<ul>
<li>
<p><strong>Bus</strong> : A true memory bus based interface, which is directly compatible with industrial
IP blocks. This requires instructions and data to be fetched over the same
interface, and also allows memory to have variable latency.</p>
</li>
<li>
<p><strong>Harvard</strong> : A simpler interface which provides seperate instruction and data
memory interfaces. These interfaces also support combinatorial read paths,
and single-cycle write paths.</p>
</li>
</ul>
<p>You need to choose one of these methods for the final submission, but might find
it useful to start with harvard and then migrate to bus. Most of the internal
control and arithmetic logic can be directly shared between the two approaches,
as long as you are taking a disciplined approach to decomposing your design.
It is also (intentionally) possible to take a Harvard CPU and wrap it in a
module which will transparently adapt it to the Bus interface, which can
be another route to a working bus-based CPU.</p>
<p>If you include both a bus and a harvard variant in your submission, then it
is assumed that you intend the bus version to be the submitted version.</p>
<p>Because the harvard version simplifies away a number of more real-world
constraints, the functionality mark is scaled by 0.8 compared to the
same functionality in a bus CPU. The other components (testing and documentation)
are unaffected by whether harvard or bus is used.</p>

<p>Both interfaces share the following common signals:</p>
<div class="snippet-clipboard-content position-relative" data-snippet-clipboard-copy-content="module mips_cpu_...(
    input logic clk,
    input logic reset,
    output logic active,
    output logic[31:0] register_v0,
"><pre><code>module mips_cpu_...(
    input logic clk,
    input logic reset,
    output logic active,
    output logic[31:0] register_v0,

<p>All signals are synchronous to <code>clk</code>, including <code>reset</code>.</p>
<p>The <code>reset</code> signal must be held high for at least 1 cycle to reset the CPU. This
is a level-sensitive reset, which is synchronous to the clock.</p>
<p>The <code>active</code> signal should be driven high when <code>reset</code> is asserted, and remain
high until the CPU halts. Once the CPU has halted (for any reason) the <code>active</code>
signal should be sent low.</p>
<p>If the CPU has completed execution (i.e. it has been reset and then <code>active</code> has been
sent low), then <code>register_v0</code> should contain the final value of register <code>$v0</code> (register index 2) from the
register file. This is purely to make your test-benches easier, and is not
something typically included in a CPU IP core.</p>
<p>The CPU does not have any support for interrupts or other input/output signals. The
only way of communicating is via memory bus transactions, the <code>active</code> signal, and
the <code>register_v0</code> signal.</p>
<h2><a id="user-content-bus-based-interface" class="anchor" aria-hidden="true" href="#bus-based-interface"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Bus based interface</h2>
<p>The CPU uses a single <a href="https://www.intel.com/content/dam/www/programmable/us/en/pdfs/literature/manual/mnl_avalon_spec.pdf" rel="nofollow">Avalon</a>
compatible memory-mapped interface to interact with memory. Your
CPU acts as a bus controller, and issues read and write transactions in order to change
memory contents. However, it is important to remember that your CPU should be completely
independent of the memory itself. The memory may be a genuine hardware RAM implemented
using BRAM or DDR, or it could be a completely virtual memory provided by a test-bench.</p>
<p>The bus-based CPU interface has the following signals:</p>

<h2><a id="user-content-harvard-interface" class="anchor" aria-hidden="true" href="#harvard-interface"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Harvard interface</h2>
<p>Everything is easier if there are two seperate instruction and data memory
buses, <em>and</em> the memory interfaces support combinatorial (zero-cycle) reads.
Taken together, these allow you to build the simple single-cycle data-path developed
during the first week of lectures. However, this is also very unrealistic, as
most CPUs (ignoring embedded micro-controllers) only have access to a single memory
bus, and have to deal with variable memory stall cycles. Unfortunately,
such a single memory bus design is complex, and represents a difficult starting
point, as there are two main ways of implementing it - either you need to
effectively implement an instruction and data cache plus appropriate
stall logic, or you need to implement a more complex multi-cycle finite-state
machine to execute the instructions.</p>
<p>The harvard interface here allows you to choose to use the simpler interface,
which removes a lot of that complexity. The interface is as follows:</p>
<div class="snippet-clipboard-content position-relative" data-snippet-clipboard-copy-content="module mips_cpu_harvard(

prefixed <code>data_</code> implement the data bus.</p>
<p>The new signal <code>clk_enable</code> supplies a clock enable, and should be
used to determine whether to update your flips-flops in a given cycle.
The general pattern for updating registers with a clock
enable is:</p>
<div class="snippet-clipboard-content position-relative" data-snippet-clipboard-copy-content="always_ff @(posedge clk) begin

<p>The interface semantics guarantee that if <code>clk_enable</code> is high then the following conditions all hold:</p>
<ol>
<li><code>instr_readdata == MEMORY[instr_address]</code></li>
<li><code>data_read==1 -&gt; data_readdata == MEMORY[data_readdata]</code></li>
<li><code>data_write==1 -&gt; MEMORY[data_address] == instr_writedata</code></li>
</ol>
<p>Note that <code>A -&gt; B</code> means logical implication, so "if A then B".</p>
<p>You should still combinatorially drive all other output signals (e.g. <code>data_read</code>, <code>data_write</code>, <code>instr_addr</code>)
during cycles where <code>clk_enable==0</code>, as the <code>clk_enable</code> signal is in part derived from
those signals.</p>
<p>The Harvard interface does not provide access to byte enables, which means
that partial store instructions (e.g. <code>sh</code>, <code>sb</code> and <code>swl</code>) are quite complicated.
If you are getting to that level it is probably better to switch to the
bus based interface.</p>
<p>Constraints on the interface are:</p>
<ul>
<li>
<p><code>! (data_read &amp; data_write)</code> : You cannot read and write in the same cycle.</p>
</li>
<li>
<p><code>data_write==1 -&gt; instr_addr != data_addr</code> : You cannot modify the instruction currently
begin read (note the comment later on self-modifying code).</p>
</li>
</ul>
<h2><a id="user-content-reset-behaviour" class="anchor" aria-hidden="true" href="#reset-behaviour"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Reset Behaviour</h2>
<p>During reset (i.e. while the <code>reset</code> signal is high), the CPU should not initiate
any memory transactions, as the memory may also be resetting at the same time.</p>
<p>The <code>reset</code> signal may be held high for more than one cycle, as other IP
cores or devices could be driven by the same reset and need more than one
cycle to reset.</p>
<p>It is not specified what the CPU should do during reset, but the
<em>effect</em> of reset should be that:</p>
<ul>
<li>All ISA-visible MIPS data registers are set to zero.</li>
<li>The next instruction to be executed post-reset should be at address <code>0xBFC00000</code>.</li>
</ul>
<p>The address <code>0xBFC00000</code> is the <a href="https://en.wikipedia.org/wiki/Reset_vector" rel="nofollow">reset vector</a>
of the CPU, and is the conventional reset vector for a "real" MIPS CPUs. The slightly
odd address is to place it at the start of the 4MB region <code>[0xBFC00000,0xC0000000)</code>.</p>
<h2><a id="user-content-cpu-halt" class="anchor" aria-hidden="true" href="#cpu-halt"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>CPU Halt</h2>
<p>Often CPUs do not "finish" in a meaningful way, and the expectation is
that once a CPU powers on there will always be work for it to do. However,
here we want a definitive end point for CPU execution, in order to make
testing more tractable - we need to know when the CPU being tested has
finished, so that we can look at how it has modified memory. To make
things easier when learning, it is also very useful to have visibility
on some internal CPU state, as doing everything via memory assumes
you already have working memory instructions.</p>
<p>To make testing easier we include the <code>active</code> flag and the <code>register_v0</code>
flag. The dual purpose of these signals is:</p>
<ol>
<li>To detect when the CPU has finished executing instructions.</li>
<li>To allow a single 32-bit value to be passed from inside the CPU to
the top-level module, without requiring any memory transactions.</li>
</ol>
<p>The CPU is considered to halt when it executes the instruction at
address 0. This behaviour is specific to this coursework specification, and
not a general property of the MIPS ISA, ABI, or commercial IP cores.</p>
<p>The reason for this choice is intimately related to the reset conditions
and <a href="https://en.wikipedia.org/wiki/MIPS_architecture#Calling_conventions" rel="nofollow">MIPS O32 ABI</a>;
in particular, this choice exploits the following existing requirements:</p>
<ul>
<li>For the reset, we require that all registers (including the PC) are set to 0.</li>
<li>The MIPS ABI also specifies that integer return values from functions are placed in register $v0, which is defined to be register 2.</li>
<li>The MIPS ABI also specifies that the return address for a function is stored in register $ra, which is defined to be register 31.</li>
</ul>
<p>This means that the following function:</p>
<div class="snippet-clipboard-content position-relative" data-snippet-clipboard-copy-content="int f(){
    return 23;
}
"><pre><code>int f(){
    return 23;
}
</code></pre></div>
<p>can be assembled into the following assembly:</p>
<div class="snippet-clipboard-content position-relative" data-snippet-clipboard-copy-content="f:  li $2, 23   # Load 23 into register $2
    jr $31      # Jump to the address in $31 (which will be zero)
    nop
"><pre><code>f:  li $2, 23   # Load 23 into register $2
    jr $31      # Jump to the address in $31 (which will be zero)
    nop
</code></pre></div>
<p>Note that a compiler is likely to exploit the delay slot, and so will
probably produce the following shorter code which exploits the delay slot:</p>
<div class="snippet-clipboard-content position-relative" data-snippet-clipboard-copy-content="f:  jr $31      # Jump to the address in $31 (which will be zero)
    li $2, 23   # Load 23 into register $2
"><pre><code>f:  jr $31      # Jump to the address in $31 (which will be zero)

</code></pre></div>

the ISA says about advancing the PC and branches.</p>

<p>Your CPU should not make any explicit assumptions about the location
of instructions, data, or peripherals within the address space. It should
simply execute the instructions it is given, and perform reads and writes
at the addresses implied by the instructions.</p>
<p>There are only two special memory locations:</p>
<ul>
<li><code>0x00000000</code> : Attempting to execute address 0 causes the CPU to halt.</li>
<li><code>0xBFC00000</code> : This is the location at which execution should start after reset.</li>
</ul>
<p>Whether a particular address maps to RAM, ROM, or something else is entirely
down to the top-level circuit outside your CPU. It may be that the top-level
is a test-bench which contains small simulated memories, and simply maps
transactions to reads and writes of a verilog array. Or the test-bench
could emulate only the specific addresses that it expects to be read or written,
without tracking the actual memory contents. Alternatively your CPU may have
been synthesised into an FPGA, in which case the memories may correspond
to a large set of block RAMs, DDR, network adaptors, and anything else
your customer decided to attach the CPU to.</p>
<h1><a id="user-content-exceptions" class="anchor" aria-hidden="true" href="#exceptions"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Exceptions</h1>
<p>Our memory bus has no mechanism for indicating that a particular
read or write access failed, in order to keep the interface simple.
This means that there is no portable way for you to test how
a given processor responds to invalid addresses. The only thing
you can do is give it test-cases which will result in it accessing
a known sequence or range of addresses, and then check that it does indeed
access those addresses. If a CPU-under-test ever accesses an
address which is outside that set of known addresses, then
you can legitimately claim that it failed the test-case, and
halt the test-bench immediately (if you wish). Similarly,
if the CPU-under-test does not access an address which you know
must be accessed, then it must also have failed.
<em>You are not required to validate the exact sequence of addresses,</em>
<em>this is simply talking about what is valid or not to test.</em></p>
<p>There is also no defined mechanism to allow CPUs to indicate
that an arithmetic exception has occurred (e.g. overflow). As
a consequence, the various overflow-checking instructions (<code>add</code>, <code>sub</code>)
etc. are not included in the testable set of instructions. So while
you can implement them in your CPU, you should not attempt to
execute them in your general test-bench. Note that <code>gcc</code> will
not generate such instructions by default, so you will not see
them if compiling C code to MIPS.
<em>This restriction is quite artificial and only for coursework purposes.</em>
<em>There is a well-defined mechanism based on exception handlers</em>
<em>that could have been used, and would require no changes to the</em>
<em>Verilog interface.</em></p>
<p>A CPU is not required to have any specific handling for undefined
or out-of-spec instructions. So a correct CPU can take any
reasonable default behaviour if it is asked to execute an instruction which
is outside the defined set of testable instructions. Note that
"reasonable" does not mean "any" - you shouldn't deliberately
take destructive actions if an invalid instruction is encountered.</p>

<p>Your test-bench is a bash script called <code>test/test_mips_cpu_bus.sh</code> or <code>test/test_mips_cpu_harvard.sh</code>
that takes a required argument specifying a directory containing an RTL CPU implementation, and
an optional argument specifying which instruction to test:</p>
<div class="snippet-clipboard-content position-relative" data-snippet-clipboard-copy-content="test/test_mips_cpu_(bus|harvard).sh [source_directory] [instruction]?

<p>Here <code>source_directory</code> is the relative or absolute path of a directory

each line containing the following components separated by whitespace:</p>
<ol>
<li>Testcase-id : A unique name for the test-case, which can contain any of the characters <code>a-z</code>, <code>A-Z</code>, <code>0-9</code>, <code>_</code>, or <code>-</code>.</li>
<li>Instruction : the instruction being tested, given as the lower-case MIPS instruction name.</li>

<p>If there is a particular package that you want to use, such as a python

run once as root before your test-bench is installed.</p>

<ul>
<li>Ubuntu package installation via <code>apt install</code>. This must be a standard Ubuntu package,
with no use of PPAs or other package sources.</li>
<li>Python package installation via <code>pip install</code> or <code>pip3 install</code>. This must be a package
coming from the standard pip set of packages.</li>
</ul>
<h1><a id="user-content-clarifying-notes" class="anchor" aria-hidden="true" href="#clarifying-notes"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Clarifying notes</h1>
<h2><a id="user-content-self-modifying-code" class="anchor" aria-hidden="true" href="#self-modifying-code"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Self-modifying code</h2>
<p>No distinction should be made between instruction and data addresses - it is legal
to both read a memory address as data and to execute it. For almost all implementations
this should happen naturally, and is a corner case that only comes into effect
with seperate instruction and data caches.</p>
<p>However, we will require that no address that is executed as an instruction
is every modified. This is because we lack any method to tell CPUs that their
instruction caches (if they exist) may have been invalidated by data accesses.</p>
<h2><a id="user-content-how-to-choose-between-bus-and-harvard" class="anchor" aria-hidden="true" href="#how-to-choose-between-bus-and-harvard"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>How to choose between bus and harvard?</h2>
<p>If you think about it, a large amount can be shared between
the two as long as you create split things up logically. In
terms of test-cases for MIPS instructions, they are going to
be the same between the two approaches. It is only the test-bench
which is going to have to implement a different interface for
the CPU, but the instructions it loads can be the same.</p>
<p>Similarly, in the CPU you should find that all the instruction
decode and execute logic is mostly the same. It is only the
parts that deal with instruction timing and memory that are
different. So you can have a single shared execution core
that is used by two variants.</p>
